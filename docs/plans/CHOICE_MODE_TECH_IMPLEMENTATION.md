# LiveTalking_v2 `choice` 模式技术实施文档

## 1. 文档目标

本文基于当前 `LiveTalking_v2` 项目中的现有实现，设计第三种对话模式 `choice`。

当前系统已有两种模式：

- `echo`：用户输入什么，数字人直接播报什么
- `chat`：用户输入文本，经 LLM 生成答复，再由 TTS 和数字人模块播报

目标是在不推翻现有主链路的前提下，新增一种结构化、可控、低时延的分支对话模式：

- 每一轮向用户展示 3 个可选问题/对话选项
- 用户选择其中 1 个选项后，数字人播报该选项对应的答复
- 每个答复节点下继续给出新的 3 个选项
- 整体形成一棵三叉对话树

本文只做技术设计，不涉及代码修改。

---

## 2. 现状分析

### 2.1 当前主链路

结合现有代码，当前项目的核心链路可以概括为：

1. 前端通过 `/offer` 建立 WebRTC 会话
2. 服务端创建 `avatar_session`
3. 前端调用 `/human`
4. `/human` 根据 `type` 分发
5. `echo` 直接调用 `avatar_session.put_msg_txt(...)`
6. `chat` 调用 `llm_response(...)`，流式产出文本片段
7. 文本片段进入 TTS 队列
8. TTS 输出音频帧
9. 数字人模块依据音频特征持续产出口型驱动视频帧
10. WebRTC 持续将音视频推给前端

### 2.2 当前系统的几个关键特点

#### 1. `/human` 当前是轻量分发器

现有路由只识别 `echo` 和 `chat`，说明新增 `choice` 最自然的入口仍然是 `/human`，或者围绕 `/human` 扩展一组配套接口。

#### 2. `chat` 当前是“自由生成”

`llm.py` 里当前是标准流式 LLM 输出，按标点分句后即时送入 TTS。因此它适合开放对话，但不适合强控制、固定分支、多节点预缓存的场景。

#### 3. TTS 链路已经具备流式能力

无论是 `GPT-SoVITS` 还是 `QwenTTS`，当前都支持流式或准流式输出，系统内部也已经按 20ms 音频块分发。因此 `choice` 模式的实时性优化重点不应只盯在数字人推理，而应重点前移到：

- 选项树的组织方式
- 答案文本生成方式
- TTS 预合成和缓存策略

#### 4. 会话对象已经是天然状态容器

当前每个用户 WebRTC 会话对应一个 `avatar_session`。因此 `choice` 模式最适合将“当前所在树节点、已选择路径、已缓存候选分支”等状态挂在 session 侧管理，而不是做成全局共享状态。

## 3. `choice` 模式的业务定义

### 3.1 业务行为

`choice` 模式定义如下：

- 会话启动后，系统先给出一个根节点
- 根节点包含：
  - 当前数字人要说的引导语
  - 3 个可选项
- 用户点击任意一个选项
- 系统进入该选项对应的子节点
- 子节点包含：
  - 数字人的答复文本
  - 下一轮 3 个可选项
- 持续重复，形成一棵三叉树

### 3.2 一个节点的最小业务单元

建议把“可播报节点”定义为：

```json
{
  "node_id": "root.a2.b1",
  "level": 2,
  "avatar_id": "wav2lip256_avatar1",
  "question_label": "你为什么这么判断？",
  "answer_text": "因为从目前的信息来看，更稳妥的做法是先确认用户的真实意图，再进入具体执行。",
  "choices": [
    {
      "choice_id": "root.a2.b1.c1",
      "choice_text": "那下一步我应该做什么？"
    },
    {
      "choice_id": "root.a2.b1.c2",
      "choice_text": "有没有更快一点的方法？"
    },
    {
      "choice_id": "root.a2.b1.c3",
      "choice_text": "这样做的风险是什么？"
    }
  ],
  "metadata": {
    "scene": "导览",
    "emotion": "neutral",
    "cacheable": true
  }
}
```

## 4. 树结构设计

### 4.1 推荐采用“节点表 + 边关系”设计

虽然业务看起来是一棵树，但在工程上不建议把整棵树直接嵌成一个巨大 JSON。更推荐：

- `nodes`：节点表，存当前节点的播报文本、层级、元信息
- `edges`：分支表，存父节点到 3 个子节点的关系

这样有几个好处：

- 节点可以独立缓存
- 子树可以按需加载
- 便于后续把三叉树升级成可配置分支数
- 便于做 A/B 测试或多套话术版本切换

### 4.2 推荐的数据结构

#### 节点对象

```json
{
  "node_id": "n_1024",
  "answer_text": "这是当前节点数字人的正式答复。",
  "tts_key": "sha1(answer_text + voice + avatar_id + tts_profile)",
  "children_ready": true,
  "children_count": 3,
  "level": 4,
  "tags": ["sales", "pricing"],
  "version": "v1"
}
```

#### 边对象

```json
{
  "parent_node_id": "n_1024",
  "choice_index": 1,
  "choice_text": "价格为什么比别家高？",
  "child_node_id": "n_2048"
}
```

### 4.3 节点来源的两种模式

建议同时支持两种树构建方式。

#### 模式 A：离线静态树

适合：

- 标准讲解
- 固定业务流程
- 导览
- 培训问答
- 销售话术

特点：

- 节点内容提前人工配置
- 质量稳定
- 最容易做缓存
- 实时性最好

#### 模式 B：动态生成树

适合：

- 半开放式多轮引导
- 希望每轮基于上下文重新生成 3 个选项

特点：

- 当前节点答复和下一层 3 个选项由 LLM 生成
- 灵活度高
- 实时性压力更大
- 需要更强的缓存和降级设计

#### 建议

第一阶段优先落地静态树或“静态树 + 少量动态补全”混合模式，不建议一开始就完全动态生成整棵树，否则时延不可控。

## 5. 模式分层设计

### 5.1 建议新增逻辑层

为了避免把 `choice` 逻辑直接塞进现有 `routes.py` 和 `llm.py`，建议增加一个独立的对话树服务层：

- `ChoiceSessionState`
- `ChoiceTreeProvider`
- `ChoiceOrchestrator`
- `ChoiceCacheManager`

### 5.2 各层职责

#### `ChoiceSessionState`

负责会话态：

- 当前节点 ID
- 已访问路径
- 最近展示过的 3 个选项
- 当前是否正在播报
- 当前预取到哪一层

#### `ChoiceTreeProvider`

负责拿树内容：

- 从本地 JSON / DB 读取静态树
- 或调用 LLM 生成下一层节点

#### `ChoiceOrchestrator`

负责一轮选择式对话的主流程：

1. 用户进入 `choice` 模式
2. 拉取当前节点
3. 返回 3 个选项给前端
4. 用户点选
5. 将对应节点答复交给 TTS/数字人
6. 同时异步预生成下一层 3 个子节点及音频缓存

#### `ChoiceCacheManager`

负责缓存：

- 文本节点缓存
- TTS 音频缓存
- 子树预取缓存
- 热门路径缓存

## 6. 接口设计

### 6.1 保守方案：继续复用 `/human`

当前 `/human` 已处理文本请求，最小改造可以继续复用，但需要扩展 `type` 和请求体格式。

#### 初始化 `choice` 模式

```json
POST /human
{
  "sessionid": "xxx",
  "type": "choice_init",
  "tree_id": "product_demo_v1",
  "avatar": "avatar_1"
}
```

返回：

```json
{
  "code": 0,
  "msg": "ok",
  "data": {
    "mode": "choice",
    "current_node_id": "root",
    "intro_text": "您好，您想先了解哪个方向？",
    "choices": [
      {"choice_id": "root.c1", "choice_text": "产品是做什么的？"},
      {"choice_id": "root.c2", "choice_text": "价格怎么计算？"},
      {"choice_id": "root.c3", "choice_text": "部署周期多久？"}
    ]
  }
}
```

#### 选择一个选项

```json
POST /human
{
  "sessionid": "xxx",
  "type": "choice_select",
  "choice_id": "root.c2",
  "interrupt": true
}
```

返回：

```json
{
  "code": 0,
  "msg": "ok",
  "data": {
    "selected_node_id": "node_price_intro",
    "answer_text": "我们的计费是按实际并发和服务模块组合计算的。",
    "choices": [
      {"choice_id": "node_price_intro.c1", "choice_text": "能举个报价例子吗？"},
      {"choice_id": "node_price_intro.c2", "choice_text": "有没有基础版？"},
      {"choice_id": "node_price_intro.c3", "choice_text": "私有化怎么收费？"}
    ],
    "audio_cache_hit": true,
    "prefetch_started": true
  }
}
```

### 6.2 更清晰的方案：拆成独立接口

更推荐新增一组接口：

- `POST /choice/init`
- `POST /choice/select`
- `GET /choice/state`
- `POST /choice/reset`

优点：

- 语义清晰
- 与 `echo/chat` 解耦
- 更方便前端区分不同模式
- 便于未来扩展树管理后台

### 6.3 前后端职责划分

建议：

- 后端负责“当前节点 + 3 个选项 + 播报任务 + 预取”
- 前端只负责展示选项、触发选择、显示当前路径和加载状态

前端不要自己推演树结构，否则状态一致性会变复杂。

## 7. 前端交互设计

### 7.1 新增第三个交互标签页

当前前端已有：

- 聊天
- 播报

建议新增：

- 选择式对话

### 7.2 `choice` 模式的前端区域

建议包含以下元素：

- 当前节点标题或引导语
- 3 个大按钮式选项
- 当前数字人正在说的话
- 当前路径 breadcrumb
- “返回上一步”
- “重新开始”
- “加载中 / 正在准备下一轮选项”

### 7.3 用户体验建议

#### 1. 选项要支持“先展示再播报”

用户点击一个选项后，不要等数字人说完才渲染下一层 3 个选项。更推荐：

1. 点击后立即高亮用户所选项
2. 后端立即返回该节点对应的下一层 3 个选项
3. 数字人开始播报本轮答案
4. 前端在右侧或下方提前显示下一轮选项，但默认禁用
5. 当前播报接近结束时解锁下一轮选项

这样会明显改善用户感知时延。

#### 2. 支持“提前打断”

沿用现有 `interrupt` 能力。用户点击下一个选项时：

- 若当前播报仍未结束，可先 `flush_talk()`
- 再切到新的目标节点

#### 3. 支持弱网态兜底

若 WebRTC 已连通，但下一轮音频未准备好：

- 先展示文本和选项
- 再延后播放声音

至少保证交互不“卡死”。

## 8. 实时性目标

### 8.1 建议的时延目标

以用户点击一个选项为起点，建议目标如下：

- 选项高亮反馈：< 50ms
- 下一节点文本和 3 个新选项返回：< 150ms
- 数字人首包语音开始：< 300ms
- 首帧口型动作可见：< 500ms

如果是完全动态生成树，这个目标会更紧张，因此必须引入缓存和预生成。

### 8.2 时延主要来自哪几段

`choice` 模式下的关键时延链路：

1. 选项节点查询
2. 答复文本生成
3. TTS 首包生成
4. 音频分块进入数字人推理
5. 视频帧推送到前端

其中最容易失控的是：

- LLM 生成下一层节点
- TTS 首包时延

所以优化重点必须围绕这两段。

## 9. 缓存设计

这是 `choice` 模式能否达到实时体验的核心。

建议采用四层缓存。

### 9.1 L1：会话内节点缓存

缓存粒度：

- 当前会话已访问过的节点内容
- 当前节点下一层的 3 个子节点
- 每个子节点的答复文本

缓存位置：

- 直接挂在 session 内存对象上

适用场景：

- 当前用户反复前进、后退
- 同一会话内重放

优点：

- 命中快
- 无序列化成本

### 9.2 L2：会话内音频缓存

缓存粒度：

- 节点答复对应的音频 PCM / WAV 数据
- 或者已经切好的 20ms 音频帧块

建议：

- 优先缓存标准化 PCM 数据
- 不建议直接缓存过多 Python 小对象帧块，容易造成内存碎片

缓存 Key：

```text
tts_cache_key = hash(
  answer_text +
  avatar_id +
  tts_engine +
  voice_id +
  ref_audio +
  tts_params_version
)
```

### 9.3 L3：跨会话共享热节点缓存

缓存对象：

- 热门树节点
- 热门答复文本
- 热门答复音频

适合放在：

- 进程内 LRU
- Redis
- 本地磁盘 KV

建议：

- 文本节点可长 TTL
- 音频缓存可中 TTL
- 若音色配置变化，应立刻失效

### 9.4 L4：启动预热缓存

对以下内容做预热：

- 根节点文本
- 根节点 3 个子节点文本
- 根节点 3 个子节点音频
- 高频路径前两层子树

这是最值回票价的一层优化，因为真实业务里前 1 到 2 层命中率往往最高。

## 10. 预生成与预取策略

### 10.1 核心思想

用户当前只会选择 3 个分支中的 1 个，但系统完全可以在他做选择前，先把这 3 个候选分支都准备好。

这比等用户点击后再临时生成，时延要小很多。

### 10.2 推荐的预取策略

#### 阶段 1：进入当前节点时

立即做 3 件事：

1. 拉取当前节点的 3 个子节点文本
2. 异步生成 3 个子节点答复
3. 异步合成 3 个子节点答复音频

#### 阶段 2：用户点击某个选项后

立即：

1. 播放已缓存的该节点音频
2. 后台继续预取这个节点的下一层 3 个子节点

### 10.3 预取深度建议

默认建议：

- 当前层：必须完整预取 3 个分支
- 下一层：只预取用户最可能进入的 1 层
- 更深层：不预取，避免指数级膨胀

否则三叉树预取会迅速爆炸：

- 深度 1：3 个节点
- 深度 2：9 个节点
- 深度 3：27 个节点

因此必须限制预取边界。

### 10.4 概率驱动预取

如果业务上有用户行为统计，可以给每个选项维护一个命中概率：

```json
{
  "choice_id": "node_1001.c2",
  "hit_probability": 0.63
}
```

按概率排序，只对 Top-N 路径做更深预取。

## 11. 文本生成策略

### 11.1 不建议每次都“全动态自由生成”

原因：

- 文案不稳定
- 下一轮 3 个选项格式可能漂移
- 首 token 时延不可控
- 很难提前缓存

### 11.2 推荐的三种生成方式

#### 方案 A：纯静态话术树

最稳、最快，适合第一期上线。

#### 方案 B：静态选项 + 动态润色答复

做法：

- 3 个选项固定
- 对应答复的主干固定
- LLM 只负责轻量润色、口语化、个性化

优点：

- 结构稳定
- 质量较好
- 可以缓存主干音频

#### 方案 C：结构化 LLM 输出

让 LLM 严格返回：

```json
{
  "answer_text": "...",
  "choices": [
    "...",
    "...",
    "..."
  ]
}
```

并在服务端加：

- JSON schema 校验
- 长度限制
- 非法输出回退

如果将来要动态生成树，必须走结构化输出，而不是自由文本。

### 11.3 话术质量建议

`choice` 模式不是开放问答，它更像“带引导的对话导航”。因此答复文本需要满足：

- 适合口播
- 句长可控
- 少专业黑话
- 尽量 1 到 3 句说完
- 最后自然衔接下一轮选择

例如：

> 我们的部署通常分为标准版和私有化版两种路径。  
> 如果你愿意，我可以继续给你讲部署周期、实施流程，或者上线前需要准备的环境。

这种答复天然适合接 3 个选项。

## 12. TTS 与音频缓存策略

### 12.1 音频缓存应优先缓存“已标准化音频”

建议缓存为：

- 16kHz mono float32 PCM

原因：

- 当前项目内部就是按该规格继续分块推音频帧
- 可以直接复用现有 `put_audio_frame` 链路
- 避免每次命中缓存后还要重新解码 WAV/OGG

### 12.2 缓存命中后的播报方式

命中音频缓存时，不再走在线 TTS 服务，而是直接：

1. 从缓存取出 PCM 数组
2. 按 20ms 分块
3. 逐块送 `avatar_session.put_audio_frame(...)`

这样首包时延会非常低。

### 12.3 冷启动时的策略

如果音频缓存未命中：

1. 先返回文本和 3 个选项
2. 并发发起 TTS 合成
3. 一旦 TTS 首包返回，立刻开始播报
4. 合成完成后写入缓存

### 12.4 音频缓存持久化建议

建议支持两种层级：

- 内存缓存：追求极速命中
- 本地磁盘缓存：避免服务重启后完全失效

磁盘目录示例：

```text
cache/
  tts/
    avatar_1/
      qwentts/
        e4/9c/e49c0c....pcm
```

## 13. 其他支持实时性和质量性的方案

除了缓存，建议同时引入以下优化。

### 13.1 方案一：根节点全量预热

服务启动后自动预热：

- 根节点引导语
- 根节点三个子节点答复
- 根节点三个子节点音频

这是最简单也最有效的低时延优化。

### 13.2 方案二：异步双通道返回

用户点击选项后，后端响应不应等待完整播报结束。应拆成：

- 控制通道：立即返回新的节点和选项 JSON
- 媒体通道：WebRTC 持续播放音频视频

当前项目已经具备这种天然结构，所以 `choice` 模式应继续沿用。

### 13.3 方案三：答复文本长度控制

建议对单节点答复设上限：

- 中文 30 到 90 字优先
- 超过阈值自动切分成两段

好处：

- 降低 TTS 合成耗时
- 减少用户等待
- 口播更自然

### 13.4 方案四：选项文案标准化

3 个选项尽量统一风格：

- 每项不超过 18 字
- 尽量是完整问题
- 语义区分明显

这样用户选择更快，误触更少。

### 13.5 方案五：并发队列隔离

建议把 `choice` 模式的后台任务拆成几个独立队列：

- 节点生成队列
- TTS 合成队列
- 缓存写入队列
- 预取任务队列

避免某个慢节点拖死整个主会话线程。

### 13.6 方案六：超时回退

如果动态生成某节点超时：

- 直接回退到预置兜底节点
- 保证用户永远还能看到 3 个选项

例如：

- “我可以从价格、功能、部署方式三个方面继续介绍，您想先看哪一个？”

### 13.7 方案七：热点路径统计

记录：

- 每个节点被访问次数
- 每个选项被点击次数
- 每条路径平均停留时长
- 每个节点缓存命中率
- 每个节点 TTS 首包时延

基于真实数据持续优化：

- 哪些路径该预热
- 哪些节点文案过长
- 哪些分支无人点击

### 13.8 方案八：前端“可交互先行”

比起“声音先出来”，用户更在意“界面先响应”。因此建议把用户体验优先级排成：

1. 点选立即高亮
2. 下一层选项立即可见
3. 文本立即可见
4. 声音尽快开始
5. 完整口型平滑输出

这能显著提升实时感知。

## 15. 推荐的执行流程

### 15.1 初始化流程

1. 前端进入 `choice` tab
2. 调用 `choice_init`
3. 服务端读取根节点
4. 返回根节点引导语和 3 个选项
5. 后台异步预取根节点 3 个子节点的答复与音频

### 15.2 用户点击选项流程

1. 前端发送 `choice_select`
2. 服务端校验该选项是否属于当前节点
3. 更新 session 当前节点
4. 若命中音频缓存，直接开始播报
5. 同时返回下一层 3 个选项给前端
6. 后台继续预取下一层的 3 个候选分支

### 15.3 异常流程

#### 子节点不存在

回退到兜底节点。

#### TTS 超时

先返回文本和选项，延迟播报。

#### 动态生成失败

回退到静态话术模板。

#### 用户连续快速点击

使用 `interrupt`，只保留最后一次选择。

## 17. 风险与难点

### 17.1 三叉树的指数膨胀

如果完全预生成，节点数会快速爆炸，因此必须限制：

- 树深
- 预取深度
- 音频缓存总量

### 17.2 动态生成的一致性

若下一层 3 个选项由 LLM 实时生成，则容易出现：

- 风格漂移
- 选项重复
- 逻辑跳跃
- 文案不适合口播

所以必须加结构校验和兜底模板。

### 17.3 音色变化导致缓存失效

当前项目支持通过 `tts.ref_file/ref_text` 动态配置音色，因此音频缓存 key 必须把音色配置纳入，否则会串音。

### 17.4 多会话并发下的内存压力

若每个会话都缓存多层音频，内存会增长很快。必须做：

- LRU 淘汰
- 会话结束时清理
- 限制单 session 缓存预算

## 20. 最终建议

### 20.1 架构建议

`choice` 模式不要只是给 `/human` 多加一个 `if`。更合理的做法是把它当成一个独立的“结构化对话子系统”：

- 复用现有数字人渲染和 TTS 主链路
- 新增独立的树管理、会话状态、缓存和预取能力

### 20.2 实施建议

优先级建议如下：

1. 先做静态三叉树
2. 再做节点级音频缓存
3. 再做当前层 3 分支预取
4. 最后再引入动态 LLM 树生成

这是性价比最高、风险最小的路线。

### 20.3 结论

从当前 `LiveTalking_v2` 的代码结构看，新增 `choice` 模式是完全可行的，而且非常适合利用现有系统已经具备的：

- WebRTC 会话管理
- `avatar_session` 状态容器
- 流式 TTS
- 音频分块驱动数字人口型

真正决定体验成败的，不是数字人渲染层，而是：

- 对话树的数据组织
- 是否把节点内容提前准备好
- 是否把音频缓存做到 session 级和热点共享级
- 是否把用户感知时延拆解为“界面先响应、声音后跟上”

如果按本文方案实施，`choice` 模式可以在较低改动风险下落地，并具备继续向更复杂的引导式、剧情式、多角色分支对话演进的能力。
