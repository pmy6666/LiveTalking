# 选项对话 EchoMimicV3 节点缓存机制任务方案计划书

生成日期：2026-04-28

## 1. 目标

在 `.` 项目的选项对话界面中，为 `echomimicv3` 模型增加节点级缓存机制。核心思路是：对所有可能被用户选择到的选项节点提前做离线预处理，生成并保存对应的音频、视频帧和元数据；用户实际点击选项后，服务端优先直接加载缓存内容并推送播放，从而显著降低 EchoMimicV3 实时生成带来的等待时延。

本计划书只描述实施方案，不修改当前代码。

## 2. 当前项目现状

项目当前已经具备选项对话基础能力：

- 前端 `web/dashboard.html` 已有“选项对话”界面。
- 前端 `web/client.js` 已调用 `/choice/init`、`/choice/select`、`/choice/state`、`/choice/reset`。
- 后端 `server/routes.py` 已提供对应的 choice 接口。
- `choice/orchestrator.py` 已有静态树加载、节点状态管理、TTS 音频缓存和子节点预取逻辑。
- `data/choice_trees/default_choice_tree.json` 使用 `nodes + choices` 的静态树结构。
- `cache/choice_audio/` 已存在 choice 节点 TTS 音频 `.npy` 缓存。
- `avatars/echomimicv3_avatar.py` 当前是整段音频进入 EchoMimicV3，生成完整视频帧后再播放。

目前已有缓存主要针对 TTS 音频。对于 `echomimicv3` 来说，真正的高延迟部分是“音频 -> EchoMimicV3 视频帧生成”。所以新缓存机制应从“音频缓存”升级为“节点级可播放片段缓存”。

## 3. 设计原则

1. 不改变选项树业务结构，仍以 `tree_id + node_id` 作为对话节点主索引。
2. 缓存必须绑定模型、角色、TTS 音色、prompt、生成参数，避免错误复用。
3. 运行时只做快速读取和播放，不在用户点击后触发 EchoMimicV3 重推理。
4. 预处理脚本可重复执行，支持跳过已命中缓存、强制重建、校验缓存完整性。
5. 第一阶段优先服务静态选项树；动态 LLM 生成节点以后再接入在线预热或后台补缓存。

## 4. 缓存对象定义

建议缓存粒度为“一个 choice node 对应一段可播放内容”。

每个节点缓存包含：

- 节点文本：`answer_text` / `tts_text`
- TTS 音频：16k、mono、float32 PCM
- EchoMimicV3 视频帧：BGR 或 RGB 帧序列
- 对齐信息：fps、音频采样率、帧数、音频 samples 数
- 生成参数：avatar_id、ref_image_path、prompt、negative_prompt、seed、sample_size、num_steps 等
- 校验信息：cache_key、schema_version、created_at、source_tree_hash

推荐目录结构：

```text
cache/
  choice_echomimicv3/
    default_choice_tree/
      <cache_key>/
        meta.json
        audio.npy
        frames.npz
        preview.jpg
```

`audio.npy` 保存 `np.float32` 的 16k 单声道音频。`frames.npz` 第一阶段可保存压缩后的帧数组；如果后续发现读盘解压慢或文件过大，可改成 `frames/000000.jpg` 或 `frames/000000.webp`。

## 5. 缓存 Key 方案

缓存 Key 必须足够严格，建议使用 SHA1 或 SHA256：

```text
cache_key = hash({
  "schema_version": "choice_echomimicv3_v1",
  "tree_id": tree_id,
  "node_id": node_id,
  "answer_text": normalized_tts_text,
  "avatar_id": avatar_id,
  "model": "echomimicv3",
  "ref_image_sha1": sha1(ref_image_file),
  "tts": tts_name,
  "tts_ref_file_sha1": sha1(ref_audio_file),
  "tts_ref_text": ref_text,
  "prompt": prompt,
  "negative_prompt": negative_prompt,
  "sample_size": echomimicv3_sample_size,
  "fps": fps,
  "num_steps": echomimicv3_num_steps,
  "guidance_scale": echomimicv3_guidance_scale,
  "audio_guidance_scale": echomimicv3_audio_guidance_scale,
  "transformer_path": echomimicv3_transformer_path,
  "weight_dtype": echomimicv3_weight_dtype
})
```

这样做可以保证：只要文本、音色、角色图、prompt 或 EchoMimicV3 参数变化，旧缓存就不会被错误命中。

## 6. 预处理脚本设计

建议新增脚本：

```text
scripts/precompute_choice_echomimicv3_cache.py
```

脚本职责：

1. 读取 `data/choice_trees/<tree_id>.json`。
2. 遍历所有 `nodes`，提取每个节点的 `tts_text` 或 `answer_text`。
3. 使用与线上一致的 TTS 参数生成音频。
4. 使用与线上一致的 EchoMimicV3 avatar、prompt 和生成参数生成视频帧。
5. 将音频、视频帧、元数据写入 `cache/choice_echomimicv3/<tree_id>/<cache_key>/`。
6. 输出预处理报告，包括成功数量、跳过数量、失败节点、总耗时、缓存体积。

建议命令形式：

```bash
python scripts/precompute_choice_echomimicv3_cache.py \
  --tree_id default_choice_tree \
  --avatar_id wav2lip256_avatar1 \
  --tts gpt-sovits \
  --force 0 \
  --max_workers 1
```

第一阶段 `max_workers` 建议固定为 1，因为 EchoMimicV3 GPU 推理本身显存占用高，并发预处理容易 OOM。

## 7. 预处理流程

完整离线流程：

```text
加载选项树
  -> 校验节点结构
  -> 计算 tree_hash
  -> 初始化 TTS
  -> 初始化 EchoMimicV3 engine 和 avatar
  -> 遍历所有可播报节点
  -> 生成 cache_key
  -> 若缓存完整且未 force，则跳过
  -> 生成 TTS 音频
  -> 生成 EchoMimicV3 帧
  -> 做音视频长度对齐
  -> 原子写入缓存目录
  -> 写入 manifest
```

原子写入建议：

```text
<cache_key>.tmp/
  -> 写完并校验成功
  -> rename 为 <cache_key>/
```

这样可以避免脚本中断后留下半成品缓存被线上误读。

## 8. 运行时加载方案

运行时需要在 `ChoiceOrchestrator` 和 `EchoMimicV3Real` 之间增加一个“节点视频缓存播放”分支。

建议逻辑：

```text
用户点击选项
  -> ChoiceOrchestrator 找到 next_node
  -> 若当前 avatar model == echomimicv3
      -> 计算 echomimicv3 节点 cache_key
      -> 命中 cache/choice_echomimicv3
          -> 调用 avatar_session.play_cached_video_segment(...)
          -> 返回 video_cache_hit=true
      -> 未命中
          -> 回退现有逻辑：TTS 音频缓存或实时 TTS + EchoMimicV3 生成
  -> 若不是 echomimicv3
      -> 继续使用现有 choice_audio 缓存逻辑
```

`play_cached_video_segment(...)` 的推荐行为：

1. 读取 `meta.json`、`audio.npy`、`frames.npz`。
2. 清空当前未播放队列，尊重 `interrupt=true`。
3. 将缓存帧和音频按当前 `fps`、`chunk=320 samples` 对齐推入播放队列。
4. 播放过程中检查 `playback_token`，用户再次点击时可立刻丢弃旧缓存播放。

## 9. 前端接口返回建议

在现有 `/choice/select` 返回值中增加缓存命中信息：

```json
{
  "selected_choice_id": "root.c1",
  "path": ["root", "capability"],
  "current": {
    "node_id": "capability",
    "answer_text": "我吃过啦，你呢？再忙也记得按时吃饭。",
    "choices": []
  },
  "audio_cache_hit": true,
  "video_cache_hit": true,
  "cache_mode": "echomimicv3_precomputed"
}
```

前端可继续用当前 `choice-note` 展示状态，例如：

- `video_cache_hit=true`：当前节点已命中数字人视频缓存，正在快速播放。
- `audio_cache_hit=true` 且 `video_cache_hit=false`：音频已命中，数字人视频仍需生成。
- 都未命中：使用实时生成回退链路。

## 10. Manifest 设计

建议额外维护一个总索引文件：

```text
cache/choice_echomimicv3/default_choice_tree/manifest.json
```

示例：

```json
{
  "schema_version": "choice_echomimicv3_v1",
  "tree_id": "default_choice_tree",
  "tree_hash": "xxx",
  "avatar_id": "wav2lip256_avatar1",
  "created_at": "2026-04-28T12:00:00+08:00",
  "items": {
    "root": {
      "cache_key": "abc123",
      "status": "ready",
      "duration_seconds": 2.4,
      "frames": 60,
      "audio_samples": 38400
    }
  }
}
```

运行时可以先查 manifest，减少重复计算和目录扫描。即使 manifest 丢失，也可以通过 `<cache_key>/meta.json` 重建。

## 11. 缓存校验与失效

缓存读取前至少校验：

- `meta.json` 存在且 JSON 可解析
- `audio.npy` 存在且 samples 数大于 0
- `frames.npz` 存在且帧数大于 0
- `meta.fps` 等于当前服务 fps
- `meta.audio_sample_rate` 等于 16000
- `meta.cache_key` 等于本次计算出的 cache_key
- `meta.schema_version` 等于当前代码支持版本

失效策略：

- 文本、音色、角色图、prompt、EchoMimicV3 参数变化：自动产生新 key，不复用旧缓存。
- 手动清理：提供 `--clean_stale` 清理 manifest 中不存在或当前 key 不再匹配的目录。
- 强制重建：预处理脚本支持 `--force 1`。

## 12. 性能目标

预期目标：

- 已命中视频缓存时，用户点击后接口返回：小于 150ms。
- 已命中视频缓存时，首帧开始推送：小于 300ms，主要受读盘和队列调度影响。
- 未命中视频缓存时，保持当前回退能力，不让用户流程中断。
- 预处理阶段允许慢，但必须稳定、可恢复、可重复执行。

需要重点观察：

- 单个节点缓存大小。
- 全树缓存总大小。
- `frames.npz` 解压耗时。
- WebRTC 播放端音视频同步。
- 打断后旧缓存片段是否会继续播放。

## 13. 分阶段实施计划

### 阶段一：缓存格式和预处理脚本

任务：

- 定义 `ChoiceEchoMimicV3CacheKeyBuilder`。
- 定义 `ChoiceEchoMimicV3CacheStore`。
- 新增 `scripts/precompute_choice_echomimicv3_cache.py`。
- 支持遍历静态树并生成 `audio.npy`、`frames.npz`、`meta.json`。
- 支持跳过已完成缓存和失败报告。

验收：

- 对 `default_choice_tree` 全部节点完成预处理。
- 每个节点缓存目录完整。
- 随机抽取节点，可离线读取帧数、音频长度和 meta。

### 阶段二：运行时命中和播放

任务：

- 在 choice 选择链路中识别 `opt.model == "echomimicv3"`。
- 命中节点视频缓存时绕过实时 EchoMimicV3 生成。
- 在 EchoMimicV3 avatar 中增加缓存片段播放入口。
- `/choice/select` 返回 `video_cache_hit` 和 `cache_mode`。

验收：

- 用户点击已预处理节点后，不触发 EchoMimicV3 GPU 推理日志。
- 数字人能直接播放缓存视频和对应音频。
- 连续点击不同选项时，旧片段能被打断。

### 阶段三：质量优化

任务：

- 对帧存储格式做压测：`npz`、JPEG 序列、WebP 序列。
- 增加 LRU 内存热缓存，优先缓存根节点和下一层节点。
- 增加预加载当前节点 3 个子节点缓存的后台任务。
- 优化 `preview.jpg` 和缓存报告，便于人工检查。

验收：

- 根节点和第一层选项几乎瞬时播放。
- 常用路径缓存命中稳定。
- 缓存体积和加载时间可控。

### 阶段四：动态节点扩展

任务：

- 对 LLM 动态生成的选项节点做后台补缓存。
- 增加“先用音频缓存播放，后台生成视频缓存，下次命中视频”的策略。
- 对热门路径做异步批量预热。

验收：

- 静态树路径全部命中。
- 动态路径第二次访问可命中缓存。

## 14. 风险与处理

### 风险一：缓存体积过大

EchoMimicV3 是完整帧缓存，体积会远大于音频缓存。应先对默认树做实测，再决定帧格式。

处理：

- 第一版使用 `frames.npz` 简化实现。
- 如果体积过大，改为 JPEG/WebP 序列。
- 对冷门节点只缓存音频，不缓存视频。

### 风险二：参数不一致导致画面不稳定

如果线上 prompt、seed、sample_size 和预处理不一致，会出现缓存画面和实时画面风格差异。

处理：

- 所有关键参数写入 `meta.json`。
- 运行时严格校验 key。
- 不允许用模糊匹配复用缓存。

### 风险三：用户打断时播放旧片段

缓存播放是快速队列推送，如果不检查 token，可能点击新选项后旧片段继续播放。

处理：

- 缓存播放必须复用现有 `playback_token` 机制。
- 每帧入队和出队前都检查 token。
- `/choice/select` 默认保持 `interrupt=true`。

### 风险四：首帧读盘仍然慢

如果缓存文件很大，点击后解压整段视频仍可能慢。

处理：

- 当前节点的三个子节点在展示选项时后台预加载。
- 支持按帧流式读取。
- 热门节点放入进程内 LRU。

## 15. 推荐验收清单

- `python scripts/precompute_choice_echomimicv3_cache.py --tree_id default_choice_tree ...` 可完整跑完。
- `manifest.json` 中 ready 节点数等于选项树 nodes 数。
- 删除任意一个节点缓存后，运行时能回退到实时生成。
- 修改节点 `answer_text` 后，重新预处理会生成新 cache key。
- 修改 `avatar_id` 或参考图后，不会误命中旧缓存。
- 连续点击三个选项，旧缓存播放不会串音、串帧。
- WebRTC 端音视频同步无明显漂移。
- 日志能区分 `video_cache_hit`、`audio_cache_hit`、`realtime_fallback`。

## 16. 最小落地版本建议

最小版本只做三件事：

1. 离线脚本遍历 `default_choice_tree`，为每个节点生成 `audio.npy + frames.npz + meta.json`。
2. 运行时 `/choice/select` 对 `echomimicv3` 优先查 `choice_echomimicv3` 视频缓存。
3. 命中后直接把缓存帧和音频推入 EchoMimicV3 的播放队列，未命中则保持当前实时生成链路。

这个版本对当前项目侵入最小，也最贴近“用户选择之后直接加载缓存内容”的目标。等最小版本跑通后，再扩展后台预加载、动态节点补缓存和缓存管理后台。
